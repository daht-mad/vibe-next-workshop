# 2.0 왜 바이브코딩인가?

> 예전 방식과 지금 방식의 차이를 이해합니다.

---

## 예전 방식: AI 서비스 투어

```
ChatGPT 열기 → 상황 설명 → 답변 복사
    ↓
Gemini 열기 → 또 상황 설명 → 답변 복사
    ↓
Claude 열기 → 또또 상황 설명 → 답변 복사
    ↓
내 작업 환경으로 돌아와서 붙여넣기
    ↓
안 되네? → 다시 AI한테 가서 설명...
```

**문제점:**
- 🔄 **컨텍스트 스위칭 지옥** - 브라우저 탭 왔다갔다
- 📋 **복붙 노가다** - 코드 복사, 에러 복사, 결과 복사...
- 🧠 **매번 처음부터 설명** - "나는 PM이고, 이 프로젝트는..."
- 🚫 **AI가 내 파일을 못 봄** - 스크린샷 찍어서 올려야 함
- ⏳ **내가 미들웨어** - AI와 작업 사이에서 전달자 역할

---

## 지금 방식: AI가 내 작업 환경에 들어옴

```
VSCode에서 작업 중
    ↓
"이거 해줘" (자연어로 말함)
    ↓
Claude가 직접 파일 읽고 → 수정하고 → 실행까지
    ↓
결과 확인 → 마음에 안 들면 "다시 해줘"
```

**이게 다른 이유:**

| 예전 (웹 AI) | 지금 (CLI 에이전트) |
|-------------|-------------------|
| AI한테 **찾아감** | AI가 내 **옆에 있음** |
| 내가 **복붙**해야 함 | AI가 **직접** 파일 수정 |
| 매번 **상황 설명** | CLAUDE.md로 **자동 기억** |
| 답변만 줌 (**조언자**) | 직접 실행함 (**실행자**) |
| 탭 **왔다갔다** | **한 화면**에서 끝 |

---

## 비유로 이해하기

**예전 방식** = 전문가한테 전화해서 물어보기
- "이거 어떻게 해요?" → 설명 듣고 → 직접 해보기 → 안 되면 다시 전화

**지금 방식** = 전문가가 옆에 앉아서 같이 작업
- "이거 해줘" → 전문가가 직접 해줌 → 맘에 안 들면 "이렇게 바꿔줘"

## 그래서 "바이브코딩"

**Vibe = 분위기, 느낌, 의도**

코드를 한 줄 한 줄 짜는 게 아니라,
**"이런 느낌으로 해줘"** 라고 의도만 전달하면 AI가 구현합니다.

```
❌ 예전: "fetch API로 GET 요청 보내고 response.json()으로 파싱해서..."
✅ 지금: "이 API에서 데이터 가져와서 표로 보여줘"
```

---

## 💡 "코딩" 안 해도 바이브코딩입니다

**바이브코딩 ≠ 코드 짜기**

여러분이 하는 일이 꼭 "코딩"이 아니어도 됩니다.

| 직무 | 바이브코딩 예시 |
|------|----------------|
| **PM** | "이 PRD 문서 읽고 유저 스토리 정리해줘" |
| **디자이너** | "이 폴더의 피드백들 분석해서 패턴 찾아줘" |
| **리서처** | "인터뷰 녹취록 5개 읽고 인사이트 뽑아줘" |
| **마케터** | "경쟁사 분석 자료 만들어줘" |
| **누구나** | "회의록 정리해줘", "보고서 초안 써줘" |

**핵심은 "코드"가 아니라 "내 작업 환경에서 AI가 직접 일한다"는 것.**

---

## 📁 개인 지식 관리에도 강력한 이유

**예전 방식의 문제:**
- 노션, 구글독스, 로컬 파일... 여기저기 흩어진 자료
- AI한테 물어보려면 → 복사해서 → 붙여넣고 → 설명하고...
- "저번에 정리했던 그거 어디 있지?" 찾기 힘듦

**바이브코딩 방식:**
```
내 폴더 구조:
📁 프로젝트/
├── 📁 리서치/
│   ├── 인터뷰-A.md
│   ├── 인터뷰-B.md
│   └── 경쟁사분석.md
├── 📁 기획/
│   └── PRD-v1.md
└── CLAUDE.md  ← Claude가 이 프로젝트를 이해함
```

```
"리서치 폴더에 있는 인터뷰 내용 종합해서 
기획 폴더에 인사이트 문서 만들어줘"
```

→ Claude가 **직접 파일을 읽고**, **새 파일을 만들어** 저장합니다.

**장점:**
- 🗂️ 내 파일 구조 그대로 활용
- 🔍 "저번에 정리한 거" Claude가 직접 찾아줌
- 📝 결과물이 **내 폴더에 파일로** 남음 (채팅창에서 사라지지 않음)
- 🔗 파일끼리 연결해서 작업 가능

## 🚀 여기에 "스킬"까지 더하면?

> Part 1에서 스킬 개념을 배웠죠? 이게 왜 게임 체인저인지 이제 이해됩니다.

> 💡 **실제 사례**: 강사가 `/billing` 명령어 하나로 96분 → 10분으로 줄인 이야기, Part 1에서 보셨죠? 이게 바이브코딩입니다.

**스킬 = 내가 정의한 업무 지침서**

반복하는 작업이 있다면? **스킬로 만들어두면 한 마디로 끝.**

```
❌ 매번: "인터뷰 녹취록 읽고, 핵심 발언 추출하고, 감정 분석하고, 
        인사이트 정리해서, 요약 문서 만들어줘. 형식은 이렇게..."

✅ 스킬: "/인터뷰분석"  ← 끝
```

**직무별 스킬 예시:**

| 직무 | 반복 업무 | 스킬로 만들면 |
|------|----------|--------------|
| **PM** | PRD → 유저스토리 변환 | `/prd-to-story` |
| **디자이너** | 피드백 정리 → 액션아이템 | `/피드백정리` |
| **리서처** | 인터뷰 → 인사이트 추출 | `/인터뷰분석` |
| **마케터** | 경쟁사 → 비교 분석표 | `/경쟁사분석` |
| **누구나** | 회의 → 회의록 + 할일 | `/회의록` |

**스킬의 힘:**

```
일반 AI 사용:
"회의록 정리해줘" → 매번 형식 설명 → 매번 다른 결과

스킬 사용:
"/회의록" → 항상 같은 형식 → 우리 팀 스타일로 → 자동 저장까지
```

| 일반 사용 | 스킬 사용 |
|----------|----------|
| 매번 프롬프트 작성 | **한 단어**로 실행 |
| 매번 형식 설명 | **일관된** 결과물 |
| 결과 품질 들쭉날쭉 | **검증된** 워크플로우 |
| 나만 사용 가능 | **팀과 공유** 가능 |

> 💡 **오늘 워크샵 목표**: 여러분만의 스킬을 만들어서 GitHub에 배포!
> 그러면 언제 어디서든 `/내스킬` 한 마디로 업무 자동화.
